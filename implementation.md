# Implementation Details

This document covers the technical implementation details of the Ternkonnect Digital Accessibility Platform.

## Technology Stack

### Frontend
- **Framework**: React 19 (via Vite)
- **Language**: TypeScript (`.tsx`, `.ts`)
- **Styling**: Vanilla CSS with CSS Variables for dynamic theming (Light/Dark mode)
- **Linting & Types**: ESLint, TypeScript Compiler

### Backend
- **Server**: Node.js, Express.js
- **Database ORM**: Sequelize (PostgreSQL)
- **Authentication**: JWT (JSON Web Tokens), bcryptjs for hashing
- **Real-Time Communications**: `ws` package for WebSockets

## Database Models

The relational database uses Sequelize to define the following core tables:

1. **User**: Stores email, hashed password, role (admin, user), and links to Organizations.
2. **Organization**: Represents the institution or school a user belongs to.
3. **IntegrationPin**: A secure 6-digit hashed PIN generated by users. It is required to authenticate external client WebSocket requests to the proxy.
4. **Subscription**: Manages the user's tier, usage limits, and expiration dates.
5. **UsageLog**: Tracks API/AI proxy usage per user for rate limiting and billing.
6. **Tool**: Represents different accessibility or educational tools available on the platform.

*Database sync is handled automatically on backend startup using `sequelize.sync({ alter: true })`.*

## REST API Endpoints

The Express server maps the following route controllers:

### 1. `auth.js` (`/api/auth`)
- **POST `/register`**: Creates a new User.
- **POST `/login`**: Authenticates a user and returns a JWT token.
- **GET `/me`**: Returns the current authenticated user's profile.

### 2. `pin.js` (`/api/pin`)
- **POST `/generate`**: Generates a new 6-digit random PIN, hashes it via `bcrypt`, and stores it in `IntegrationPin`.
- **GET `/list`**: Lists all active pins for the logged-in user.
- **DELETE `/:id`**: Revokes/deletes a pin.

### 3. `tools.js` (`/api/tools`)
- **GET `/`**: Retrieves a list of all active accessibility tools.
- **POST `/usage`**: Logs a usage event to the `UsageLog` table.

## The WebSocket Proxy Architecture

The defining feature of the backend is the `/api/tools/proxy` WebSocket server.

**Why a Proxy?**
To use Google's Gemini Bidi (Bidirectional) Generative API, you need a secret API key. Exposing this key to the frontend is a massive security risk. The proxy allows the frontend to stream audio/text directly to Google Gemini while the backend injects the API key securely.

**How it works (Implementation Level):**
1. An Express HTTP Server is created using `http.createServer(app)`.
2. A separate WebSocket server `const wss = new WebSocket.Server({ noServer: true })` is initialized.
3. The server listens to the `upgrade` event. When a request comes to `/api/tools/proxy`, it extracts the `email` and `pin` from the query parameters.
4. It compares the plaintext `pin` against the hashed pins in the `IntegrationPin` table using `bcrypt.compare`.
5. If valid, the connection is upgraded to a WebSocket.
6. The backend opens a secondary WebSocket connection to `wss://generativelanguage.googleapis.com/...` appending the master `GEMINI_API_KEY` from the environment.
7. Event listeners `.on("message")` are piped between `clientWs` and `targetWs`, allowing full-duplex communication without exposing credentials.

## Frontend Architecture

The frontend is a lightweight Single Page Application:
- **`App.tsx`**: Main entry component. Integrates Ternkonnect branding, loads routing components, and manages root state.
- **`index.css`**: Defines CSS variables (`--primary`, `--accent`, `--bg`, etc.) controlling the light/dark modes and maintaining the Ternkonnect brand guidelines.
- **Hooks & Contexts**: Custom React hooks manage WebSocket connections, voice commands, TTS (Text-to-Speech), and high-contrast settings seamlessly.
